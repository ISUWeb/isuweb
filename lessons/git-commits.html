# Git - Commits

In this lesson we will learn the basics of Git, a tool for saving versions of our code.

Git keeps track of changes to our code as it evolves over time, allowing us to save snapshots of our code at any point. This is increadibly usefult if we want to revert changes to make our code look exacly as it did at a previous point in time. This also creates a history of all changes, giving us the oportunity to document why they were made. Not only that, we can have multiple versions of the code that exist in parallel so that we (or perhaps many people) can work on making different changes at once.

We will begin this lesson by saving a snapshot of how our code looks at the moment. This is known as a commit that we can come back to if we ever want our code to look as it does now. We will save this snapshot both on our local machines and on the cloud (GitHub).

When we create a commit, we want it to include changes made to this file. To do this we need to add it to the group of files that make up our commit. To do this is to add the file to the staging area. You might see a warning about line endings. This is okay, but take a moment to research why if you're curious.

We now want to save our repository on GitHub so that we have a backup and so that others can get our changes.

To do this we need to tell git where to send the data. This is done using a url, which we get after creating a repository on GitHub. It looks something like https://github.com/your-username/your-repo.git.

However, we don't want to type this everytime we want to send our changes to GitHub. So let's create an alias for this, which by convention is origin. From now on, origin and that url will mean the same thing.

Don't worry if you don't see any output after running it. That's normal.

¬†


~/git-demo> git remote add origin https://github.com/your-username/your-repo.git
~/git-demo>
